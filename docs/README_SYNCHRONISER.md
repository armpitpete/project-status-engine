# Daily README Synchroniser

## Purpose

The daily README synchroniser copies validated completion results from the trusted internal owner dataset into explicitly marked README sections.

It is a consumer of completion authority. It does not calculate percentages, infer readiness, choose authority or inspect repository activity to estimate progress.

## Trusted input

The only completion input is:

```text
internal-build/completion-status.json
```

The dataset is generated by `project-status-engine` during the same trusted workflow run. It is not read from the public Pages output or the private top-five dashboard.

## Owner-wide rollout boundary

The production policy is declared in:

```text
config/readme-sync-policy.json
```

`mode: all-valid` selects every discovered active repository whose internal completion record has state `valid`. Repository names and URLs are never committed to the policy or generated allowlist; targets are represented only by SHA-256 identifiers inside the trusted runner.

The original three contract-shaped selectors now live at `tests/fixtures/readme-sync-contract-shapes.json`. They are test-only regression fixtures and are not production configuration or an allowlist.

Repositories without a valid `.project/progress.json` are not assigned 0%, estimated, or written. They remain outside the synchroniser until a human-approved authority contract exists. A newly valid repository joins the target set automatically on the next run.

## README contract

A target README must contain exactly one pair of markers:

```markdown
<!-- AUTO:PROJECT-COMPLETION:START -->
<!-- AUTO:PROJECT-COMPLETION:END -->
```

Only bytes between these markers may change. All bytes before the opening marker and from the closing marker onward are verified unchanged.

Missing, duplicated or reversed markers close the complete run without a write. Marker onboarding is a separate reviewed repository change and is never performed implicitly by the synchroniser.

## Write contract

For every selected project, the synchroniser:

1. preflights the complete selected set before any write;
2. reads the default-branch README;
3. consumes the stage percentages already present in the internal dataset;
4. prepares the replacement marker block;
5. does nothing when the default-branch README already matches;
6. reads an existing synchroniser branch when one reviewable PR is already open;
7. creates no branch update or commit when that open PR already contains the desired README;
8. otherwise resets or creates only `automation/readme-sync` from the default-branch head;
9. commits only `README.md` on that automation branch;
10. opens or updates one ready-for-review pull request.

The synchroniser never writes directly to the default branch. Repeated daily runs with unchanged authority do not manufacture replacement commits on an already-current open pull request.

## Fail-closed conditions

The complete selected batch closes before writing when:

- the internal dataset is missing or malformed;
- the rollout policy is missing or invalid;
- no valid completion target exists;
- selected completion is invalid;
- authority is absent;
- stages or supplied percentages are malformed;
- README markers are absent, duplicated or reversed;
- more than one open synchroniser pull request exists for a target;
- an open synchroniser pull request has no readable automation branch;
- repository or API state cannot be read safely.

Repositories whose completion state is not valid are excluded before the write batch is constructed. This is not treated as 0% completion.

## Privacy and credential controls

- repository identities are resolved only inside the trusted runner;
- the committed rollout policy and test fixtures contain no repository identity;
- logs contain only fixed action names and opaque target digests;
- API errors are reduced to fixed operation/status codes;
- tokens, URLs, response bodies, authority paths, stage labels and percentages are not logged;
- pull-request title and body are fixed generic text;
- one target README receives only its own completion record;
- neither the internal dataset nor the resolved target allowlist is written under `public/` or uploaded to Pages.

A dedicated `README_SYNC_TOKEN` is required for owner-wide application. It must have repository access to every project intended for synchronisation, with Contents and Pull requests set to read and write.

## Schedule

The workflow runs daily at 05:17 in `Europe/London`. It can also be invoked manually.

Pull-request and ordinary push events run deterministic validation only; they do not perform cross-repository writes.
